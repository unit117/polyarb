This audit identifies several critical accounting and logic errors that explain the system's current performance (high unrealized "gains" but negative realized PnL and a 0.4% win rate).

## Critical (will lose money or corrupt state)

### 1. Directional Error in Implication Constraints
- **What**: The system assumes `Market A` always implies `Market B`, regardless of which threshold is higher.
- **Where**: `services/detector/constraints.py` (Lines 58-62) and `_check_price_threshold_markets`.
- **Why**: `_implication_matrix` hardcodes `matrix[0][1] = 0` (A=Yes implies B=Yes). However, `build_constraint_matrix` does not reorder markets so that the "stronger" condition is Market A. If Market A is "BTC > $70k" and Market B is "BTC > $80k", the system will incorrectly conclude that $70k implies $80k.
- **Fix**: In `_check_price_threshold_markets`, explicitly identify which market is the "Sub-condition" and which is the "Super-condition". Return them in a specific order or include a `direction` flag in the result to be used by `build_constraint_matrix`.

### 2. Broken Short Position Accounting
- **What**: `execute_trade` does not handle "BUY" orders as covering existing short positions.
- **Where**: `services/simulator/portfolio.py` (Lines 40-52).
- **Why**: When `side == "BUY"`, the code always increases `positions` and adds to `cost_basis` as if opening a long. If the current position is `-100` (short) and you buy `50`, the position becomes `-50`, but the `cost_basis` (which for shorts tracks credit received) is *increased* by the cost of the buy-back. This corrupts the cost basis and PnL calculation.
- **Fix**:
```python
if side == "BUY":
    current = self.positions.get(key, Decimal("0"))
    if current < 0: # Covering a short
        abs_pos = abs(current)
        avg_credit = self.cost_basis[key] / abs_pos
        self.cost_basis[key] -= min(size_d, abs_pos) * avg_credit
    else: # Opening/increasing long
        self.cost_basis[key] = self.cost_basis.get(key, Decimal("0")) + size_d * price_d
    self.positions[key] = current + size_d
```

### 3. Missing Realized PnL Tracking in Portfolio
- **What**: `execute_trade` never updates `self.realized_pnl` or `winning_trades`.
- **Where**: `services/simulator/portfolio.py` (Lines 31-90).
- **Why**: Realized PnL is only calculated in the `SimulatorPipeline` wrapper or during `close_position` (settlement). If a position is partially closed via `execute_trade`, the PnL is "lost" from the metrics, though it remains in the `cash` balance. This makes the dashboard `Realized PnL` metric completely incorrect for rebalanced trades.
- **Fix**: Move the PnL calculation logic from `pipeline.py` (Lines 112-126) directly into `portfolio.py:execute_trade` so the portfolio object is the single source of truth for accounting.

### 4. Price Threshold Timestamp Mismatch
- **What**: Logical implication is incorrectly applied to markets with different time windows.
- **Where**: `services/detector/classifier.py` (Lines 186-198).
- **Why**: If Market A has a timestamp (e.g., "at 10AM") and Market B does not (e.g., "Daily"), the check `if start_a and start_b` fails. The code then proceeds to compare thresholds and returns an `implication` dependency. A daily high of $80k does *not* imply the price was > $70k at 10AM.
- **Fix**:
```python
# Line 186
if (start_a or start_b) and (start_a != start_b or end_a != end_b):
    # If either has a timestamp and they don't match exactly, they are independent
    return {"dependency_type": "none", ...}
```

## High (likely causes incorrect behavior in production)

### 1. Erroneous Simplex Projection in Frank-Wolfe
- **What**: Manually projecting to a simplex after the Frank-Wolfe update.
- **Where**: `services/optimizer/frank_wolfe.py` (Lines 105-106).
- **Why**: Frank-Wolfe is a projection-free algorithm; the update `q = (1-gamma)q + gamma*s` is a convex combination of two points already in the marginal polytope. Projecting `q` to a simplex *after* the update can actually move the point *outside* the marginal polytope (violating logical constraints) while satisfying only the sum-to-1 constraint.
- **Fix**: Remove lines 105-106. If numerical stability is a concern, clip values to `[0, 1]` and re-normalize, but do not use simplex projection which ignores the logical constraints `s` was solved for.

### 2. Portfolio Restoration "Flip" Bug
- **What**: Rebuilding cost basis fails when a single trade flips a position from Long to Short (or vice versa).
- **Where**: `services/simulator/main.py` (Lines 68-75).
- **Why**: The logic checks `if pos > 0` (position before trade). If a SELL trade of 100 shares is applied to a +20 share position, the entire 100 shares are treated as "reducing a long," resulting in a negative cost basis.
- **Fix**: Split the trade into two parts: the portion that closes the existing position (to 0) and the portion that opens the new side.

### 3. Invalid Partition Matrix for Multi-Outcome Events
- **What**: `_partition_matrix` forces exactly one of two markets to be Yes.
- **Where**: `services/detector/constraints.py` (Lines 78-82).
- **Why**: This is only true for binary partitions (A vs Not A). In a 3-outcome event (A, B, C), if Market 1 is "A" and Market 2 is "B", it is perfectly feasible for both to be "No" (if C is "Yes"). The current matrix `[[0, 1], [1, 0]]` makes `A=No, B=No` infeasible.
- **Fix**:
```python
if n_a == 2 and n_b == 2:
    # Only force exactly-one if we are sure it's an exhaustive binary partition
    # Otherwise, only A=Yes + B=Yes is infeasible.
    return [[0, 1], [1, 1]] 
```

## Medium (suboptimal, could mask problems)

### 1. Recursive Fee Calculation Bug
- **What**: `max_size` calculation uses the original fee instead of a proportional one.
- **Where**: `services/simulator/portfolio.py` (Line 44).
- **Why**: If `cost > self.cash`, `max_size` is calculated by subtracting the *original* `fees_d` (calculated on a larger size). This results in a smaller `max_size` than the portfolio can actually afford.
- **Fix**: `max_size = self.cash / (price_d * (1 + self.fee_rate))` (assuming fee is a percentage of notional).

### 2. Dangerous Fallback in PnL Calculation
- **What**: Defaulting cost to 0.5 when cost basis is missing.
- **Where**: `services/simulator/portfolio.py` (Line 149).
- **Why**: If the system loses track of cost basis (due to the restoration bugs above), it assumes an entry price of 0.5. This will hallucinate PnL and corrupt the `total_value` and `unrealized_pnl` metrics.
- **Fix**: Log a critical error and return 0.0 for PnL if cost basis is missing; never guess the entry price.

## Suggestions (improvements, not bugs)

### 1. Tighten Conditional Constraints
- **What**: The "Positive Correlation" logic in `constraints.py` (Lines 133-145) is extremely aggressive.
- **Why**: It turns a 15% price divergence into a hard logical impossibility. This is likely why the system is "detecting" arbitrage in correlated markets that are simply lagging or have different liquidity profiles.
- **Fix**: Increase the `DIVERGENCE_THRESHOLD` to 0.30 or remove the hard `0` constraints for conditional types, relying instead on the KL-divergence to pull them together softly.

### 2. Fix Iteration Counter
- **What**: Redundant ternary.
- **Where**: `services/optimizer/frank_wolfe.py` (Line 111).
- **Fix**: `iterations = t + 1`.
