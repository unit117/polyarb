## Critical (will lose money or corrupt state)

### Broken Short-Covering Accounting
- **What**: `execute_trade` BUY side fails to realize PnL or correctly update cost basis when covering a short position.
- **Where**: `services/simulator/portfolio.py` lines 42–56
- **Why**: The BUY logic only handles increasing a long position. If `self.positions[key]` is negative (short), it simply adds the new shares and adds `size * price` to the `cost_basis`. For shorts, `cost_basis` tracks credit received. Adding `size * price` (the cost to cover) to the credit received is mathematically incorrect and results in an orphaned, inflated cost basis that never realizes PnL.
- **Fix**: Check if `current < 0`. If so, calculate realized PnL as `(avg_entry - vwap_price) * shares_covered`, reduce the absolute `cost_basis` proportionally, and update `self.realized_pnl`.

### Desync on Position Flips (Long to Short)
- **What**: `execute_trade` SELL side caps execution at the current long position size, preventing position flipping.
- **Where**: `services/simulator/portfolio.py` line 59
- **Why**: `sell_size = min(size_d, current) if current > 0 else size_d`. If you are long 100 shares and sell 150, `sell_size` becomes 100. The remaining 50 shares are discarded, but the returned `result` and the `PaperTrade` persisted in the DB will reflect the full 150. This creates a permanent desync between the DB and the in-memory portfolio.
- **Fix**: Allow `sell_size` to exceed `current`. If `size_d > current`, close the long position (0 to `current`) and open a short position for the remainder (`size_d - current`).

### Zero-Division Crash on Portfolio Restore
- **What**: Startup crash when restoring a portfolio that has perfectly closed positions.
- **Where**: `services/simulator/main.py` line 78
- **Why**: During cost basis reconstruction, `avg = portfolio.cost_basis[key] / pos` is calculated. If a trade perfectly closed a position, `pos` (which is `current_pos + trade_size`) will be 0 if the trade being processed is the one that opened the position (this logic is generally inverted). More simply, if `pos` is 0, the service crashes.
- **Fix**: Add a guard: `if pos != 0: avg = ...`.

### Missing Collateral Modeling for Shorts
- **What**: Simulator allows "double-spending" cash that should be locked as collateral for short positions.
- **Where**: `services/simulator/portfolio.py` and `pipeline.py`
- **Why**: When selling short, the simulator adds `proceeds` to `self.cash` (line 58). In reality, Polymarket/Kalshi lock the maximum potential obligation (usually $1/share) as collateral. By making the proceeds immediately available for other trades, the simulator overestimates available liquidity, allowing the system to take on massive, unbacked risk.
- **Fix**: Subtract the maximum obligation ($1.00 * abs(shares)) from `available_cash` when calculating trade capacity in `pipeline.py`.

## High (likely causes incorrect behavior in production)

### Kelly Sizing Units Error
- **What**: Kelly sizing is applied to a fixed dollar cap instead of total equity, resulting in tiny, fee-negative trades.
- **Where**: `services/simulator/pipeline.py` lines 127–142
- **Why**: `kelly_fraction` (e.g., 0.025 for a 5% edge) is multiplied by `max_position_size` ($100). This results in a `base_size` of 2.5 *shares*. At a price of $0.50, the trade cost is $1.25. The fixed taker fees ($0.08–$0.10) eat ~8% of the trade, which is larger than the 5% edge.
- **Fix**: `base_size` should be `(kelly_fraction * total_equity) / price`.

### Double-Counting Fees in Execution Gate
- **What**: Execution gate subtracts fees from an edge that already had fees subtracted.
- **Where**: `services/simulator/pipeline.py` line 192
- **Why**: `estimated_profit` from the optimizer already accounts for venue fees. The pipeline then checks `if post_vwap_edge - per_share_fee <= 0: break`. This double-deduction causes the system to reject valid, profitable arbs.
- **Fix**: Remove the fee subtraction in `pipeline.py` or only subtract the *difference* between estimated and actual VWAP fees.

### Pass 1 Cash Reservation is Overly Conservative
- **What**: Arbs are blocked if the BUY leg exceeds current cash, even if a SELL leg in the same arb provides the necessary funds.
- **Where**: `services/simulator/pipeline.py` lines 174–184
- **Why**: `reserved_cash` only tracks BUY costs. It does not track SELL proceeds. In a balanced arb (Sell A, Buy B), the system should be able to execute even with $0 starting cash.
- **Fix**: Track `net_cash_flow` in Pass 1. `net_cash_flow += proceeds` (SELL) and `net_cash_flow -= cost` (BUY). Block only if `self.portfolio.cash + net_cash_flow < 0` at any step.

### Incomplete Multi-Outcome Partition Constraints
- **What**: `_partition_matrix` for multi-outcome markets allows logically impossible joint outcomes.
- **Where**: `services/detector/constraints.py` lines 100–105
- **Why**: It marks `oa != ob` as infeasible (0). However, it defaults all other cells to feasible (1). In a partition, if `oa == ob`, then `Market A = Yes` **requires** `Market B = Yes`. The current matrix allows `Market A = Yes` and `Market B = No` for the same outcome, which is a logical violation.
- **Fix**: Initialize matrix to 0. Set `matrix[i][j] = 1` only if `(oa == ob)` or if the outcomes are truly independent within the partition.

## Medium (suboptimal, could mask problems)

### Stale Price Snapshots
- **What**: 120-second snapshot age is too high for volatile prediction markets.
- **Where**: `shared/config.py` line 52 (`max_snapshot_age_seconds`)
- **Why**: Executing an arbitrage strategy on 2-minute-old data in a high-frequency environment (like crypto "Up or Down" markets) will lead to high slippage and "ghost" arbs that have already vanished.
- **Fix**: Reduce `max_snapshot_age_seconds` to 10–15 seconds.

### Suboptimal Multi-Outcome Trade Selection
- **What**: `compute_trades` only selects the single best leg per market.
- **Where**: `services/optimizer/trades.py` line 68
- **Why**: Frank-Wolfe may find that multiple outcomes in a multi-candidate market are mispriced. By only picking one, the system leaves money on the table and fails to fully "hedge" the logical gap.
- **Fix**: Allow multiple trades per market if they all contribute to reducing the Bregman gap.

## Suggestions (improvements, not bugs)

### Rebalancing Win-Rate Tracking
- **What**: Rebalancing exits (closing positions before settlement) are not counted in win-rate metrics.
- **Where**: `services/simulator/pipeline.py` line 224
- **Why**: Only `close_position` (settlement) updates `winning_trades`. This makes the dashboard win-rate misleading.
- **Fix**: Call `self.portfolio.mark_settled(is_winner=realized > 0)` inside the pipeline's rebalancing block.

### Restore Performance
- **What**: `_restore_portfolio` queries the entire `paper_trades` table on every restart.
- **Where**: `services/simulator/main.py` line 55
- **Why**: As the trade history grows to 100k+ rows, this will cause significant startup delays.
- **Fix**: Only query trades where `executed_at > latest_snapshot.timestamp`.
