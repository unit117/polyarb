## Critical (will lose money or corrupt state)

### 1. Portfolio Restore Logic double-counts trades and corrupts cost basis
- **What**: The `_restore_portfolio` function initializes positions from a snapshot but then replays **all** historical trades to rebuild cost basis, leading to incorrect accounting.
- **Where**: `services/simulator/main.py`, lines 40–90
- **Why**: If a snapshot exists, it already reflects the state of all trades up to that timestamp. Replaying every trade from the beginning of the `paper_trades` table on top of a snapshot-initialized position will double-count the impact of trades on the cost basis. Furthermore, line 73 calculates `pos` using the *current* position (from the snapshot) to determine the cost basis of a *historical* trade, which is mathematically nonsensical.
- **Fix**: Only query and replay trades where `executed_at > latest.timestamp`.

### 2. Frank-Wolfe normalization hides internal market arbitrage
- **What**: The optimizer normalizes market prices to sum to 1.0 before processing, which masks "trivial" arbitrage within a single market (overrounds/underrounds).
- **Where**: `services/optimizer/frank_wolfe.py`, line 85 and `_normalize` function (lines 135–140).
- **Why**: If a binary market is priced at Yes: 0.60 / No: 0.60 (1.20 total), there is a 20% arbitrage. By normalizing these to 0.50 / 0.50, the optimizer sees a "fair" market and reports zero theoretical profit, failing to capture the most basic form of arbitrage.
- **Fix**: Remove `_normalize` and project the raw price vectors. The Bregman projection will naturally find the nearest point in the marginal polytope, and the distance will correctly reflect both internal and cross-market inconsistencies.

### 3. Kelly sizing uses incorrect formula and fixed dollar cap
- **What**: The Kelly sizing logic uses a unitless "edge" as a fraction and scales against a fixed `$100` cap instead of total equity.
- **Where**: `services/simulator/pipeline.py`, lines 127–142
- **Why**: Kelly is defined as `f* = (bp - q) / b`. The code uses `edge * 0.5`. If the edge is 0.05 (5%), it calculates a `kelly_fraction` of 0.025. It then multiplies this by `max_position_size` ($100), resulting in a $2.50$ trade. This is a massive under-bet. Conversely, if the portfolio grows to $100k, it still bets $2.50$. If it drops to $100, it still bets $2.50$ (which is now 2.5% of equity, potentially over-leveraged).
- **Fix**: Calculate `base_size` as `kelly_fraction * total_value`. Ensure `kelly_fraction` uses the proper `(edge / price)` calculation for binary outcomes.

## High (likely causes incorrect behavior in production)

### 1. In-flight deduplication allows duplicate trades on the same pair
- **What**: The system tracks in-flight trades by `opportunity_id` instead of `pair_id`.
- **Where**: `services/simulator/pipeline.py`, lines 46–56
- **Why**: The detector generates a new `opportunity_id` every time it runs (e.g., every 60s). If an execution for `pair_id: 500` is slow (waiting for VWAP or DB), and the detector runs again, it creates a new `opportunity_id` for the same pair. The simulator will see this as a "new" opportunity and execute the same arbitrage again, doubling the intended exposure.
- **Fix**: Use `self._in_flight: set[int]` to store `pair_id` instead of `opportunity_id`.

### 2. Partition matrix assumes exhaustive outcomes
- **What**: The binary partition matrix hardcodes a "one-and-only-one" constraint.
- **Where**: `services/constraints.py`, lines 106–110
- **Why**: For binary markets, it returns `[[0, 1], [1, 0]]`. This forces exactly one "Yes" and one "No". If the markets are "Trump wins" and "Harris wins" in a race with other candidates, they are a partition (mutually exclusive) but **not exhaustive**. Both resolving "No" is a feasible outcome. This matrix would incorrectly flag "Both No" as an arbitrage opportunity.
- **Fix**: Only use `[[0, 1], [1, 0]]` if the markets are explicitly identified as an exhaustive pair (e.g., A vs !A). Otherwise, use a standard mutual exclusion matrix `[[0, 1], [1, 1]]`.

### 3. Double-counting fees in PnL metrics
- **What**: Realized PnL calculation in the pipeline subtracts fees that were already deducted from cash.
- **Where**: `services/simulator/pipeline.py`, lines 259–262
- **Why**: `portfolio.execute_trade` already reduces `self.cash` by `size * price + fees`. The pipeline then calculates `realized = (exit_price - avg_entry) * close_size - exit_fees` and adds it to `realized_pnl`. This results in the fee impact being recorded twice: once in the cash balance and once in the PnL metric, causing the "Total Value" to be reported incorrectly.
- **Fix**: Realized PnL should be `(exit_price - avg_entry) * size`. The fees are a separate transaction cost already reflected in the cash balance.

## Medium (suboptimal, could mask problems)

### 1. VWAP fee calculation uses unscaled size
- **What**: When a trade is reduced due to insufficient capital, the fee is not recalculated.
- **Where**: `services/simulator/portfolio.py`, lines 44–48
- **Why**: `max_size` is calculated using the `fees_d` from the *original* requested size. If a user requests a $10,000 trade (fee $150) but only has $500 cash, the `max_size` will be calculated as `(500 - 150) / price`, which is significantly smaller than the actual affordable size.
- **Fix**: Recalculate `fees_d` inside the `if cost > self.cash` block based on the `max_size`.

### 2. Price Threshold regex misses "reach" and "hit" verbs
- **What**: The price threshold detector only looks for "above/below/over/under".
- **Where**: `services/detector/classifier.py`, line 144 (`_PRICE_THRESHOLD_RE`)
- **Why**: Polymarket frequently uses questions like "Will Bitcoin **reach** $100,000?" or "**hit** $100,000?". These are logically identical to "above", but the current regex will skip them, forcing them into the slower/less reliable LLM classifier.
- **Fix**: Update regex to: `r"^(.+?)\s+(?:above|below|over|under|reach|hit|exceed)\s+\$?([0-9,]+(?:\.\d+)?)"`.

### 3. Worst-case payoff proof ignores hedge ratios
- **What**: `_worst_case_payoff` assumes a 1:1 ratio for all trade legs.
- **Where**: `services/optimizer/trades.py`, lines 173–210
- **Why**: The function iterates through trades and sums PnL assuming 1 unit per leg. However, the optimizer might suggest buying 1.0 units of Market A and 0.5 units of Market B to maintain a hedge. The "proof" will pass or fail based on the wrong weights.
- **Fix**: Pass the optimal weights `q` from the FW result into `_worst_case_payoff` and multiply the per-leg PnL by the corresponding weight.

## Suggestions (improvements, not bugs)

### 1. Stale Snapshot rejection in Optimizer
- **What**: The optimizer processes opportunities based on `detected_at` but doesn't check if the underlying prices have changed since detection.
- **Where**: `services/optimizer/pipeline.py`
- **Fix**: Add a check to ensure the `PriceSnapshot` used for optimization is not older than `max_snapshot_age_seconds`.

### 2. Implication Direction logic for "Below" markets
- **What**: The logic for "below" threshold implications is correct but less intuitive than "above".
- **Fix**: Add a comment in `constraints.py` line 235 explaining that for "below" markets, the *lower* threshold is the antecedent (e.g., "Below 50" implies "Below 100").
