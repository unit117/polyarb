## Critical (will lose money or corrupt state)

### 1. Incorrect Outcome Ordering Assumption in Constraint Matrix Generation
- **What**: Constraint matrix generation assumes a fixed order for market outcomes (e.g., `outcomes[0]` is "Yes").
- **Where**:
    - `services/detector/constraints.py`: lines 50 (`_implication_matrix`), 59-62 (`_partition_matrix` binary), 77 (`_mutual_exclusion_matrix`), 93-94, 100-101, 107-118 (`_conditional_matrix`), 140-141, 145-146, 150-151 (`_compute_profit_bound`).
- **Why**: The `Market` model stores `outcomes` as a `JSONB` (Python `dict`), where the order of keys is not guaranteed to be `["Yes", "No"]` or any specific sequence. If `outcomes_a[0]` does not correspond to the "Yes" outcome (or the intended primary outcome), the generated constraint matrices will be fundamentally incorrect. This will lead the Frank-Wolfe optimizer to find "arbitrage" based on false constraints, resulting in trades that are not true arbitrage and will likely lose money.
- **Fix**: Modify `build_constraint_matrix` and its helper functions to explicitly map outcome labels (e.g., "Yes", "No") to their corresponding indices in the `outcomes_a`/`outcomes_b` lists, rather than relying on positional indexing. This requires passing the actual outcome labels (e.g., `market_a.outcomes.keys()`) and then finding the index of "Yes" or "No" within that list.
    ```python
    # Example fix for _implication_matrix (similar changes needed for others)
    def _implication_matrix(n_a: int, n_b: int, outcomes_a_labels: list[str], outcomes_b_labels: list[str]) -> list[list[int]]:
        matrix = [[1] * n_b for _ in range(n_a)]
        if n_a >= 2 and n_b >= 2:
            # Find index of 'Yes' and 'No' outcomes, assuming they exist
            yes_a_idx = outcomes_a_labels.index("Yes") if "Yes" in outcomes_a_labels else 0 # Fallback if not found
            no_b_idx = outcomes_b_labels.index("No") if "No" in outcomes_b_labels else 1 # Fallback if not found
            # A=Yes implies B=Yes, so A=Yes + B=No is infeasible
            matrix[yes_a_idx][no_b_idx] = 0
        return matrix
    ```
    This requires `build_constraint_matrix` to accept `outcomes_a_labels` and `outcomes_b_labels` as explicit arguments, derived from `market_a.outcomes.keys()`.

### 2. `mark_settled` Misuse Leading to Inflated Trade Metrics
- **What**: The `mark_settled` method is called for partial position exits, incorrectly incrementing `settled_trades` and `winning_trades`.
- **Where**: `services/simulator/pipeline.py`: line 111 (`self.portfolio.mark_settled(is_winner=realized > 0)`).
- **Why**: `mark_settled` is designed to record a *fully settled/closed trade* for win-rate tracking. When a trade partially closes an existing position (e.g., rebalancing), the PnL for that portion is realized, but the overall position is not fully settled. Calling `mark_settled` here inflates `self.portfolio.settled_trades` and `self.portfolio.winning_trades`, leading to a completely inaccurate win rate reported in the dashboard and persisted in `PortfolioSnapshot`. The `daily-report-2026-03-21.md` already shows a "Very low win rate (0.4%): Only 12 winning trades out of 2,969 evaluated," which is likely a symptom of this bug.
- **Fix**: Remove the call to `self.portfolio.mark_settled` from `simulate_opportunity`. The `settled_trades` and `winning_trades` counters should *only* be incremented when a position is fully closed, which `portfolio.close_position` already handles correctly.

### 3. Incorrect Portfolio Restoration for Cost Basis of Short Positions
- **What**: The `_restore_portfolio` function does not correctly rebuild the `cost_basis` for short positions or for closing long positions from `PaperTrade` history.
- **Where**: `services/simulator/main.py`: lines 55-59.
- **Why**:
    - The logic `pos = portfolio.positions.get(key, Decimal("0")) + Decimal(str(t.size))` is incorrect. When replaying a `SELL` trade from `PaperTrade`, `t.size` is positive. If `portfolio.positions.get(key)` is the *current* net position (after all trades up to the snapshot), adding `t.size` to it does not represent the position *before* this specific `SELL` trade.
    - More importantly, `PaperTrade` records `t.side == "SELL"` for both closing long positions and opening/increasing short positions. The current `_restore_portfolio` logic only attempts to *reduce* `cost_basis` proportionally (for closing longs). It completely misses the case where `t.side == "SELL"` represents an *opening short*, where `cost_basis` should be *increased* by the credit received (`t.size * t.vwap_price`).
    - This will lead to an incorrect or missing `cost_basis` for all short positions, making `unrealized_pnl` and `positions_in_profit` calculations unreliable for a significant portion of the portfolio.
- **Fix**: The `_restore_portfolio` function should either:
    1. Replay *all* `PaperTrade` records sequentially from a blank portfolio state (after loading initial capital), using the `portfolio.execute_trade` method for each trade. This is the most robust way to reconstruct `positions` and `cost_basis`.
    2. Revise the `cost_basis` reconstruction logic to correctly differentiate between opening shorts, closing shorts, opening longs, and closing longs, and apply the appropriate `cost_basis` updates for each. For `SELL` trades, if `portfolio.positions.get(key)` was already negative (or zero before this trade), it's an opening/increasing short, and `cost_basis` should be increased by `t.size * t.vwap_price`.

### 4. Dashboard Displays Incorrect "Unrealized Win Rate" Metric
- **What**: The dashboard's "Unrealized Win Rate" calculation is mathematically incorrect.
- **Where**: `services/dashboard/web/src/components/StatsBar.tsx`: lines 17-20.
- **Why**: The code calculates `(p.unrealized_pnl / p.total_value) * 1000` and labels it as "unrealized win rate". A win rate is typically defined as `(number of positions in profit / total number of open positions)`. The current formula is a scaled percentage of overall unrealized PnL relative to total portfolio value, which is not a win rate and provides misleading information about the performance of individual open positions.
- **Fix**: The `Portfolio` class already provides `positions_in_profit()` which returns `(in_profit, total_positions)`. The dashboard should use this method to calculate and display the true unrealized win rate.
    ```typescript
    // In StatsBar.tsx
    // ...
    const unrealizedWinRate =
      p && p.total_positions > 0
        ? `${((p.positions_in_profit / p.total_positions) * 100).toFixed(1)}%`
        : "\u2014";
    // ...
    ```
    (Note: `p.positions_in_profit` needs to be exposed in the `PortfolioSnapshot` dict and passed to the frontend.)

### 1. Incorrect PnL and Profitability Calculation for Short Positions
- **What**: The `unrealized_pnl` and `positions_in_profit` methods in `Portfolio` have incorrect logic for short positions.
- **Where**: `services/simulator/portfolio.py`:
    - `unrealized_pnl`: lines 107-108.
    - `positions_in_profit`: lines 119-120.
- **Why**:
    - **`unrealized_pnl`**: The fallback `cost = shares * Decimal("0.5")` (line 107) is problematic for shorts (`shares < 0`). If `cost_basis` for shorts tracks credit received (a positive value), then `shares * 0.5` (a negative value) is an inconsistent fallback. The PnL calculation `shares * current - cost` (line 108) is also incorrect for shorts. For a short, PnL is `credit_received - cost_to_close`, or `cost - (abs(shares) * current)`.
    - **`positions_in_profit`**: The check `if shares * current > cost:` (line 120) is incorrect for shorts. For a short position, `shares` is negative. `shares * current` will be negative. `cost` (credit received) is positive. A negative number will never be greater than a positive number, so short positions will always be incorrectly reported as "not in profit". A short is in profit if `cost` (credit received) is greater than `abs(shares) * current` (cost to close).
- **Fix**:
    - **`unrealized_pnl`**:
        ```python
        # In Portfolio.unrealized_pnl
        # ...
        for key, shares in self.positions.items():
            current = Decimal(str(current_prices.get(key, 0)))
            cost = self.cost_basis.get(key)
            if cost is None:
                logger.warning("missing_cost_basis", key=key, shares=float(shares))
                # Fallback should be consistent with how cost_basis is tracked
                # For longs, cost is total paid. For shorts, cost is total credit received.
                cost = abs(shares) * Decimal("0.5") # Assuming 0.5 as average entry/exit price for missing data
            
            if shares > 0: # Long position
                pnl += shares * current - cost
            else: # Short position
                pnl += cost - abs(shares) * current
        return float(pnl)
        ```
    - **`positions_in_profit`**:
        ```python
        # In Portfolio.positions_in_profit
        # ...
        for key, shares in self.positions.items():
            current = Decimal(str(current_prices.get(key, 0)))
            cost = self.cost_basis.get(key)
            if cost is None:
                logger.warning("missing_cost_basis_for_profit_check", key=key, shares=float(shares))
                cost = abs(shares) * Decimal("0.5") # Fallback
            
            if shares > 0: # Long position
                if shares * current > cost:
                    in_profit += 1
            else: # Short position
                if cost > abs(shares) * current: # Credit received > cost to close
                    in_profit += 1
        return in_profit, total
        ```

### 2. Overly Permissive Structural Check for "Partition" Dependency Type
- **What**: The structural verification for `partition` dependency type is too weak, allowing invalid pairs to pass.
- **Where**: `services/detector/verification.py`: lines 51-60.
- **Why**: The condition `if len(outcomes_a) > 2 or len(outcomes_b) > 2: return True` (line 58) is too permissive. A partition implies that the union of outcomes from both markets forms an exhaustive and mutually exclusive set for a single event space. If one market is binary (e.g., {Yes, No}) and the other is multi-outcome (e.g., {A, B, C}), they cannot form a partition. This check allows such invalid combinations to pass, leading to wasted optimization cycles and potentially incorrect constraint matrices.
